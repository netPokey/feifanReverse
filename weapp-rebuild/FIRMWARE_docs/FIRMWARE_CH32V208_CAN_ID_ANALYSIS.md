# CH32V208 蓝牙 CAN 收发器固件 — CAN ID 分析报告

> 分析对象：`c6eec2f9-firmware.bin`（62 464 字节 / 0xF400，裸二进制，无符号表）
> 方法：RISC-V (RV32IMAC) 反汇编 + bxCAN 外设寄存器识别 + 常量表定位，并与本仓库
> `handlers.h` / `TESLA_CAN_*_REFERENCE.md` 交叉印证。

## 0. 固件性质确认

| 事实 | 证据 |
|------|------|
| 芯片：WCH CH32V208（青稞 V4，RV32IMAC） | 复位向量 `j 0x0800d034`；字符串 `CH32V20x_BLE_LIB_V1.3` |
| Flash 基址 `0x08000000`，gp=`0x27ffc000` | 复位处 `auipc/addi gp`；外设引用 `0x40006xxx` |
| 蓝牙协议栈：WCH BLE Lib | 同上字符串 |
| CAN 外设：bxCAN（兼容 ST），CAN1=`0x40006400` | 见下方过滤器/收发函数 |

> 注意：本仓库**源码是 ESP32-S3 版本**（`platformio.ini` = espressif32），而这颗 bin 是
> **CH32V208 版本**——两者是同一“特斯拉 CAN 控制器”产品的不同 MCU 实现。ID 逻辑高度一致，
> 故可用仓库文档为 bin 里的 ID 命名。

## 1. 识别出的 bxCAN 库函数（裸机，地址即证据）

| 地址 | 函数 | 作用 |
|------|------|------|
| `0x0800d526` | `CAN_FilterInit` | 写 `FMR/FS1R/FM1R/FFA1R/FA1R` 与过滤器组 `0x240/0x244` |
| `0x0800d628` | `CAN_Transmit` | 取空邮箱，`StdId<<21` 写 `TIxR(0x180)`，DLC/数据写 `TDLR/TDHR` |
| `0x0800d6fc` | `CAN_Receive` | 读 `RIxR(0x1b0)>>21` 取 StdId，读数据并释放 FIFO |

应用层 CAN 驱动模块在 `0x08008062 ~ 0x080082xx`：
- `0x08008062` **发送封装**：先调 `0x080001d4`（“是否允许发送”门禁）→ 为 0 直接跳过；
  ID 取自帧结构体偏移 0（`lhu 0(frame)`）。**发送 ID 由调用方/蓝牙指令决定，不是写死的。**
- `0x08008116` **按 ID 表装载过滤器**：把 ID 数组拷进缓冲（≤112B=56×u16），配置
  **14 组 × 每组 4 个 ID 的 16-bit 列表模式**（`id<<5` 正是 bxCAN 标准帧 STID 位置）。
- `0x080080ce` **全通过滤器**（14 组 mask=0，接收所有 ID 的嗅探模式）。
- `0x0800825e` **接收中断**：`CAN_Receive` → 把 `{id,dlc,data}` 存入 12 字节步长的 RAM 表
  （`0x08008218`）→ 经函数指针转发（推测发往蓝牙）。

## 2. 【收集 / 分析】的 CAN ID

### 2.1 内置监控 ID 表 @ `0x0800f188`（53 个有效 ID + `0x7ff` 哨兵）

这是固件硬编码的“已知 / 解码白名单”。表中每个 16-bit 字都是合法 11-bit 标准帧 ID（强证据）。

```
0x082 0x102 0x103 0x118 0x129 0x132 0x1f9 0x20c 0x229 0x238 0x243 0x249 0x257 0x25a
0x25d 0x266 0x273 0x292 0x293 0x2b6 0x2e1 0x2e5 0x2f3 0x312 0x31f 0x321 0x332 0x333
0x334 0x339 0x33a 0x352 0x39d 0x3b3 0x3b6 0x3c2 0x3c3 0x3d2 0x3d8 0x3df 0x3e2 0x3e3
0x3e9 0x3f5 0x3fd 0x3fe 0x401 0x405 0x4e2 0x4e3 0x4f3 0x678 0x679
```

### 2.2 信号解码分发器 @ `0x08009e2e`（二分查找，61 个 ID）

每个 ID 命中后跳到对应的解码例程提取信号。覆盖 2.1 全表，并多出 8 个
（注入/事件/扩展用）：`0xa9 0x189 0x21c 0x37a 0x3a1 0x3b0 0x4a8 0x68c`。

### 2.3 带含义的 ID（仓库文档/常见特斯拉 Model 3/Y 车身总线）

### 2.4 硬件过滤策略
- 有“全通嗅探”（`0x080080ce`）与“按表白名单”（`0x08008116`）两套配置；
- 过滤器表也可**经蓝牙动态下发**（`0x08009a2e` 调用，ID 数组来自 BLE 缓冲）——
  手机端可自定义要采集的 ID 集合。

## 3. 【发送 / 注入】的 CAN ID

固件**不是周期性广播一组固定 ID**，而是三类发送，且都受“CAN_TX 总开关”门禁
（`0x080001d4`，默认只听，和仓库 README 的 LISTEN_ONLY 安全策略一致）：

1. **拦截—改写—原 ID 回注（re-sign 注入）**
   读到特定帧 → 改数据位 → **重算特斯拉序列计数器（byte9 高 nibble）+ CRC 校验字节
   （查表 `~0x0800eadc`）** → 用**原 ID**重发。
   - 注入/事件触发 ID（flash 短表 `0x0800f17a` + 分发器 `0x08007f8e`）：
     `0x189 0x21c 0x334 0x3a1 0x3b0 0x3e9`
   - 解码器 `0x0800a4e4` 明确把 `0x3b0(944)`、`0x4a8(1192)` 交给注入处理函数 `0x08007e40`。
   - 该模块共约 6~7 个改写处理函数（`0x08007c16/0x08007c7c/0x08007cfa/0x08007d3e/
     0x08007d58/0x08007dc0/0x08007ef4`），全部尾调用发送封装 `0x08008062`。
   - 仓库 ESP32 版同款逻辑：`processIsaChimeSuppression()` 改 `0x399` 数据位并重算校验后回注。

2. **蓝牙→CAN 透传**：BLE 主机下发“ID+数据”，固件在 `0x08009a80` 处用 BLE 缓冲构帧并
   `CAN_Transmit`——**可发任意 ID**（这就是“蓝牙控制收发器”的本质）。

3. 另有若干子系统各自构帧发送（共 18 处调用 `0x08008062`）。

## 4. 一句话结论
- **收集/分析**：固件内置 **53 个监控 ID（@0x0800f188）/ 解码器覆盖 61 个**，以 BMS-电池、
  动力总成、车身/UI 信号为主；硬件过滤支持“全通嗅探”或“白名单”，且白名单可蓝牙下发。
- **发送**：默认只听；可对 `0x189/0x21c/0x334/0x3a1/0x3b0/0x3e9/0x4a8` 等帧做
  “改写+重算计数器/CRC+原 ID 回注”，并可经蓝牙透传**任意 ID**。

## 5. 安全提示
该固件能在车辆总线上回注/伪造帧（含 BMS/驱动相关 ID），**只应在隔离测试台、且确认
CAN_TX 关闭时上电观察**；未经验证不要接入真实车辆总线。
