# CH32V208 蓝牙 CAN 控制器 — 蓝牙控制协议深度分析（小程序 × 固件 联合逆向）

> 配套文档：`FIRMWARE_CH32V208_DEEP_ANALYSIS.md`（固件侧自底向上反汇编）、
> `FIRMWARE_CH32V208_CAN_ID_ANALYSIS.md`（CAN ID 表）。
> 本文从**小程序侧（`weapp-rebuild/`）自顶向下**还原蓝牙控制面，并与固件反汇编结论
> **逐条对照、互证、纠偏**，给出一份可直接复刻的“手机 ↔ 模块”完整协议规范。
>
> 证据标注：固件侧给**地址**（如 `0x080097cc`）；小程序侧给**文件:行**（如 `services/ble.js:191`）。
> 可信度：**✓双证**=固件与小程序互证；**◎程序**=仅小程序明确；**?**=仅二进制行为/待确认。

---

## 0. TL;DR — 联合逆向的 5 个关键结论

1. **固件的“命令码 `a1`” = 小程序 BLE 帧的 `type` 字段。** 固件 RE 只认出 `0xD0/0xC0/0xC1/0xBB`
   等少数几个；小程序给出**完整命令字典 `160–240`（0xA0–0xF0）**，是固件分发器 `0x080097xx`
   的地面真值。（§4）
2. **固件分析里“`0xD0` 打包 29B 状态包”被精确命名：** 就是小程序的 **`208`（0xD0）电池概览**，
   29 字节、字段位宽（SOC 10bit / 电压 / 温度）与 `batteryParser.js` **逐位吻合**。（§6.2）
3. **固件结论“蓝牙命令面无鉴权”需要修正：** 协议层存在 **4 字节蓝牙密码握手 `168`（默认 `1234`）**，
   且 app 把所有指令挡在鉴权之后。固件 RE 未识别该 handler——很可能因被分析的 `.bin` 是
   **嗅探/调试变体**。（§5）
4. **固件的“改写+原 ID 回注”被赋予语义：** 小程序的 `167/187/162` **动作码**（座椅/车门/灯光/
   AP/喇叭…共 200+ 个）就是固件 re-sign 注入（`0x333/0x3b0/0x4a8/0x4f3` 等）的上层触发源。（§8）
5. **固件的“raw CAN 透传 / `0xFF 0xFF` 改过滤器”在本消费级小程序里被裁掉了**——量产 app 只暴露
   经 `commandSafety.js` 白名单收敛过的高层动作，不暴露任意 ID 注入。（§4.3 / §10）

## 1. 方法与三者关系

### 1.1 自顶向下 vs 自底向上

| 方向 | 材料 | 能拿到什么 | 拿不到什么 |
|------|------|-----------|-----------|
| 自底向上（已完成） | `firmware.bin` 反汇编 | 收发/过滤/解码/注入的**地址与算法** | 命令码的**业务语义**、payload 字段名 |
| 自顶向下（本文） | 小程序 JS（`services/*` + `pages/*`） | **完整命令字典 + 精确 payload 位图 + 鉴权/权限/OTA** | 命令在 MCU 内部的**真实执行路径**（需回到固件） |

二者在“**BLE 帧 `type` = 固件命令码 `a1`**”这一点对齐后即可互证。

### 1.2 三者关系（重要语境）

- **firmware.bin（被分析对象）**：WCH **CH32V208** + `CH32V20x_BLE_LIB_V1.3` + bxCAN。固件字符串
  实测仅 198 条，含 `CH32V20x_BLE_LIB_V1.3`、`TSL6`（见复现）。该 bin 形态偏**嗅探/分析**
  （监控 53+ CAN ID、带 raw-CAN 与动态过滤器命令）。
- **小程序（`weapp-rebuild/`）**：产品线“**TSL 电子模块**”的官方控制端，按硬件版本 `hw_ver`
  区分多种形态（见 §3）。命令面以**高层动作**为主，**不**暴露 raw-CAN。
- **关系判定**：二者**共享 BLE 帧格式与命令码空间**（`0xD0/0xC0/0xC1/0xBB` 数值一致），属
  **同一产品线**的固件端与控制端；被分析的 `.bin` 是其中偏诊断的一个 MCU/形态变体。
  因此小程序可为固件命令**补全语义**，但个别低层命令（raw-CAN/过滤器）只在 bin 里、不在 app 里。

> 复现固件字符串：`strings -n 5 firmware.bin | grep -iE 'CH32|TSL'` → `CH32V20x_BLE_LIB_V1.3`、`TSL6`。
> 注意云端域名 `www.iffrc.com` **只在小程序里**（`services/api.js:2`），固件不做 HTTP，符合预期。

## 2. 蓝牙链路层（GATT + 帧格式）

### 2.1 GATT 画像（`services/ble.js`）

| 项 | 值 | 证据 |
|----|----|------|
| 目标 Service | `0000FFF0-0000-1000-8000-00805F9B34FB`（短 UUID `FFF0`） | `ble.js:74,115` |
| 读写/通知特征 | `0000FFF1-0000-1000-8000-00805F9B34FB`（`FFF1`） | `ble.js:83,116` |
| 写方式 | 优先 `writeNoResponse`，否则 `write` | `ble.js:83,202` |
| 通知 | `notifyBLECharacteristicValueChange`（`FFF1`） | `ble.js:83-90` |
| MTU | 动态，默认 23；分片大小 = `min(mtu,247)-3`，下限 20 | `ble.js:118,193` |
| 扫描过滤 | 仅 `services:["FFF0"]`（`targetOnly`） | `ble.js:147` |

> FFF0/FFF1 是一颗常见的“透传”BLE 模组配置（与 WCH BLE Lib 相符）。固件侧通知出口为
> `ble_notify(conn,type,buf,len)`（`0x08000c02`），与此特征一一对应。

### 2.2 应用层帧格式（双向同构）★

`services/ble.js` 同时实现了**发送封帧**与**接收拆帧**，二者用**同一帧结构**：

```
┌────┬────┬──────┬────────┬────────┬───────────────┬──────────┐
│0x55│0x7F│ type │ lenHi  │ lenLo  │   payload[len]│ checksum │
└────┴────┴──────┴────────┴────────┴───────────────┴──────────┘
  85   127   命令码    长度(大端 u16)        负载         校验
```

- **发送**（`ble.js:191-192`）：`c=[0x55,0x7F,type,len>>8,len&0xff, ...payload]`；
  `checksum = (Σ c[2..end]) & 0xFF`（从 `type` 起累加），push 到帧尾，再按 MTU 分片逐包写。
- **接收**（`ble.js:129`）：状态机 `0(找0x55)→1(找0x7F)→2(取type)→3(取lenHi)→4(取lenLo)→
  5(收 payload)→6(校验)`；`verify` 同样从 `type` 起累加；`len>4096` 直接丢弃；包间隔 ≥300ms
  复位（防粘包跨帧）。校验通过才回调 `onPacket(type, payload)`。

> **与固件互证**：固件 GATT 写回调把 `命令码 a1 / payload s1 / 长度 s2 / 连接上下文 s0` 送入
> 分发器（`FIRMWARE_..._DEEP_ANALYSIS.md` §6）。那里的 `a1` 正是这里的 `type`，`s1/s2` 正是
> 这里拆出的 `payload/len`。**`0x55 0x7F` 同步头 + 末字节和校验**是该协议的稳定指纹。

|------|--------|------|---------|
| classicVehicle | 3,5,6,7,8,19 | 车辆控制固件 | 车控/仪表/快捷/设置/备份/电池/RGB/蓝牙按钮 |

### 4.2 上行（设备→手机）分发（`app.js:90 dispatchPacket`）

```
type ∈ {160,161,163} → device.parseDeviceInfo()   // 设备信息/设置回读
type == 168          → handleBleKeyResponse()      // 鉴权结果

type == 167          → 车控回执（如座椅记忆 保存成功/失败）

type == 176          → handleGaugePacket() → gaugeParser

```

### 4.3 固件有、但小程序**不暴露**的低层命令

固件 RE（§6.1）还列了两条**诊断级**命令，**未**出现在 `commandSafety.js` 白名单里：

- **`0xFF 0xFF + idlist`** → 重设 RX 过滤白名单（`filter_load` `0x08008116`）。
- **帧 `[idHi,idLo,dlc,data…]`（len≤8）** → 透传发送**任意 CAN ID**（`tx_wrapper` `0x08008062`）。

→ 量产小程序用**收敛后的动作码（167/187/162）**取代了“任意 ID 注入”，把危险面藏进固件、
不在 app 暴露。这与“被分析 bin 偏嗅探/调试形态”一致。

## 5. 鉴权与三级权限模型（修正固件“无鉴权”结论）★★

固件 RE 结论为“蓝牙命令面无鉴权（命令码即明文）”。小程序证据表明**协议层是有鉴权的**，
且权限分三级：

### 5.1 设备级：蓝牙密码握手 `168`（✓程序，固件 RE 漏识别）

```
连接成功 → app 主动触发 handleBleKeyResponse([255])           // app.js:205
        → 设备回 168,[255]（请求密码）
手机 → tx(168, [c0,c1,c2,c3])  4 字节 ASCII（默认 "1234"）     // app.js:227
设备 → 168,[1]=正确 / [0]=错误 / [254]/[255]=需要(重)输入      // app.js:209-222
正确后设备补发 160 设备信息帧                                  // mockBle.js:200-202
```
- 失败有**本地限速**：1 小时内 3 次，超限锁定（`app.js:279 getBleKeyRemain`）。
- 校验成功后 `ble_key` 落本地缓存，下次自动重放（`app.js:212,216`）。
- 设备侧 `mockBle.js:198-203` 复刻了“**未鉴权则拒答、回 `168,[255]`**”的行为——说明
  **设备端确实按密码门控指令**。

> **纠偏**：固件 RE 未在 `0x080097xx` 找到 `0xA8(168)` 分支，故误判“无鉴权”。两种解释皆可成立：
> ① 被分析的 `.bin` 是嗅探/调试变体，鉴权被关闭（与 RE 看到的 `CAN_TX` 门禁默认 LISTEN_ONLY 一致）；
> ② RE 覆盖不全。无论哪种，**协议定义了鉴权**，量产固件应实现 `168`。建议回到 bin 复核：在分发器
> 二分/switch 里补找立即数 `0xA8` 的分支。

### 5.2 设备级：工厂/维护权限（`isAdmin`）

`commandSafety.js:97` 对 **`192` 调试监听**强制要求 `adminLevel`（工厂模式）。`165` 重启、
`167,255` 维护、`168` 改密等也按“来源页 + 二次确认”收紧（§7）。

### 5.3 App 级：访问模式（纯客户端）

`services/accessMode.js`：密码 **`tsl@admin0826` → `full`** 模式，本地存储、70 天有效
（`accessMode.js:1,41`）。它**只控制小程序 UI 可见性与自动化开关**（`canRunAutomation`
= `full`，`app.js:269`），**不下发任何 BLE 指令**，与设备鉴权相互独立。

| 层级 | 凭据 | 校验方 | 作用域 | 证据 |
|------|------|--------|--------|------|
| App 访问模式 | `tsl@admin0826` | 小程序本地 | UI/自动化可见性 | `accessMode.js:1` |
| 工厂/维护 | `isAdmin` 标志 | 命令安全层 | 高危命令(192…) | `commandSafety.js:97` |
| 设备蓝牙密码 | 4B（默认1234） | **设备固件** | 所有指令执行 | `app.js:227`/`mockBle.js:198` |

> **与固件互证**：这些字段正是固件“信号状态表 `signal_state`（`0x08000272`，12B/条）”里被
> `decode_dispatch`（`0x08009e2e`）按 CAN ID 写入的解码结果（车速 `0x118/0x257`、车门
> `0x102/0x103`、转向 `0x129`、功率 `0x266/0x2e5`、温度/续航 `0x312/0x33a`…）。小程序给出了
> 固件“状态表→蓝牙打包”那一步的**确切位布局**。

### 6.3 电池概览帧（`208`/0xD0 应答，**29B**）★★ — 与固件 §6.3 精确对接

## 8. CAN 注入模型：动作码 → 固件 re-sign 回注 ★★

这是“蓝牙控制”的核心：手机发**高层动作码**，固件把它翻译成**对特斯拉总线的帧注入**。

### 8.1 三条注入通道

| 命令 | 语义 | 码空间 | 来源 |
|------|------|--------|------|
| `167` | **直接车控**（即时） | 自有表：座椅/车门/车身/电池加热… | `vehicleControl.js` data、`commandSafety s[]` |
| `162` | **模拟滚轮**（AP 操作） | `1..6`（上/右/下/左/确认/长按） | `vehicleControl.js:82`、`commandSafety:55` |
| `187` | **执行已配置动作** | `fun` 表 `1..255`（200+ 项） | `functionOptions.js fun` |

### 8.2 `167` 直接车控表（从 `pages/vehicleControl/vehicleControl.js:24-102` 实测）

| 码 | 动作 | 码 | 动作 |
|----|------|----|------|
| 26/27/28/29 | 主驾 前移/后移/靠背前/靠背后 | 12/13/14/15 | 副驾 前移/后移/靠背前/靠背后 |
| 52/54/56 | 主驾 记忆1/2/3 | 60/58 | 主驾 女王/露营 |
| 62/64/66 | 副驾 记忆1/2/3 | 70/68 | 副驾 女王/露营 |

| 255 | 进入维护/服务模式（`my.js`/`set` 页，二次确认） | | |

> `commandSafety.js:70` 限定 `167` 只能从**首页常用入口/车辆控制页/仪表页/维护入口**发，且每个
> 入口各有**码白名单**（如首页只允许 `c=[4,5,6,7,8,9,22,23]`：锁/解锁/门盖/充电口/电池加热）。

### 8.3 `187` 动作字典（`functionOptions.js` 的 `fun` 列）

200+ 个动作，分组涵盖：座椅控制/座椅记忆/车门控制/辅助驾驶/后视镜/档位/电池&充电/上锁解锁/
车辆控制/停止模式/灯光/雾灯/喇叭/多媒体/模块控制/4G/外接 LED 屏。关键特例（`mockBle.js:212` 复刻）：

- `252`=模块启用、`251`=停用、`250`=启用/停用切换、`253`=重启（`functionOptions.js` “模块控制”）。
- `118/119`=摄像头 关/开（仪表盘低速自动摄像头，`gauge.js:596`，受 `commandSafety:89` 二次门控）。

`btn` 列则是**触发事件**字典（方向盘/滚轮/车窗/车门/换挡/盲区…），用于 `171` 把**物理按键事件**
绑定到 `fun` 动作（快捷指令），`186` 把**外接蓝牙按钮**绑定到 `fun` 动作。

### 8.4 回到固件：动作码如何变成总线帧

- 固件对 `0x333/0x4f3` 的解码 thunk **尾跳 `tx_wrapper`**（`0x08008062`），对 `0x3b0/0x4a8`
  走注入入口 `0x08007e40`，模式统一为：**改数据位 → `frame[9]+=0x10`（4bit 滚动计数器）→
  `frame[0xA]=CRC_table[counter>>4]`（查表 `~0x0800eadc`）→ 原 ID 回注**（DEEP_ANALYSIS §5.2）。
- 因此：手机 `167/187/162` 动作码 → 固件选定目标 CAN ID + 数据位 → **重算计数器/CRC** →
  注入特斯拉总线。小程序提供了“动作码→语义”，固件提供了“语义→具体帧/重签名”。两边拼起来即
  **完整控制链**。

> ⚠️ 注入受固件 `CAN_TX` 门禁 `0x080001d4` 约束（默认 LISTEN_ONLY）。真正能动车，需固件该门禁开启。

## 10. 蓝牙主机桥接 `240`（设备当 BLE 中心）

`services/bleHostProtocol.js` + `mockBle.F`（`mockBle.js:120`）揭示：模块本身可作 **BLE Central**，
去搜/连/管外接蓝牙按钮（如“方向盘按钮/左门按钮/后备箱按钮”）。

- 动作码（`bleHostProtocol.js:4`）：`FIND=1 / STOP=2 / QUERY_BOND=4 / CONNECT=5 / DELETE=7`。
- 帧前缀 **14B 设备 UID**（`getUid` 取 `[4..18]`）+ 动作码 + 参数（MAC/地址类型/名称）。
- 上行 `240` 复用同 `type`：搜索结果(`[3]…`)/配对列表(`[4]…`,尾哨兵 `90`)/状态(`[6]…`)。
- `commandSafety.js:114` 对 `240` 校验**严格**：长度、UID 非空、动作码白名单、CONNECT/DELETE 的
  MAC 与地址类型，全部校验。

> 这是固件“CAN 收发器”之外的**第二个无线角色**：既是被手机连的 Peripheral（FFF0/FFF1），又能作
> Central 桥接外设。固件 RE 未覆盖该路径（它聚焦 CAN），此处为**纯小程序侧新增**结论。

## 11. OTA 升级链路（云端 + BLE 写）

固件 RE 未涉及 OTA；小程序 `services/firmwareUpdate.js` + `services/api.js` 补全：

- **云端**：`https://www.iffrc.com/ffkj/user/ble.php`（域名可在“我的-服务器域名”改，`api.js:1`）。
  接口：`fun=getinfo`（设备元信息）、`fun=updatehelp`（升级说明）、`fun=update`（分页下载固件 HEX）。
- **取版本**：先 BLE `160/163` 帧取 `id/vin/bt_ver/fw_ver`，再 `getinfo` 用云端版本覆盖
  （`firmwareUpdate.js` `OFFICIAL_RUNTIME_HINTS.ipaChain`）。
- **固件包校验**（`firmwareUpdate.js:149 _()`）：每页 = `[2B 大端页号][数据]`；
  逐页 `CheckCode = (Σbytes)&0xFFFF`；校验页号连续、页大小一致、总长 == `BytesCount`；
  整包 FNV-1a 指纹。
- **BLE 写帧**：诊断态明确“**不发送 BLE `85/83/87/69`、不调 `updateok`**”
  （`firmwareUpdate.js:275,2`）——即真正烧写用 `type ∈ {85,83,87,69}` 这组命令 + `updateok` 收尾。
  > 注意 `85=0x55` 与帧同步头同值，但此处是 `type` 语义，非同步头；OTA 是另一组命令族。
- **门禁**：云端对**设备 id/VIN** 有版本/权限门控；开发占位 id(`001122…`)/VIN(`LRW3E7FS0NC0…1`)
  会被识别为非授权设备而拒发固件（`firmwareUpdate.js:35 i()`）。

> 安全提示：OTA 的**真实写帧族（85/83/87/69 + updateok）**是改写 MCU Flash 的入口，风险最高；
> 本仓库小程序刻意只做**只读诊断**（`buildDiagnosticPayload`），未实现写入动作。
