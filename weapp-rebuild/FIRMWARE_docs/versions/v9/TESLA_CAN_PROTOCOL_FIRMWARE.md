# 特斯拉 CAN 协议逆向（固件实证）— ID/Payload 含义 · 解析算法 · 控制注入

> 目标：**以固件 `firmware.bin` 为主**，逐一还原特斯拉 CAN ID 的 payload 含义、固件**怎么解析**收到的

> **✔FW**=固件反汇编实证；**✔APP**=小程序解析器佐证；**◎DBC**=特斯拉 Model3/Y 社区 DBC 交叉命名（未必逐位精确）；**?**=待确认。

## 1. 解析总架构（数据怎么从 CAN 流到状态）✔FW

```
CAN_RX_ISR(0x0800825e) → rx_store(0x08008218)  // 把收到的帧存成 12 字节结构
        帧结构(12B): [ID:u16 @0][DLC @2][D0..D7 @3..10][pad @11]      ← §1.1 实证
        │
        ▼
decode_dispatch(0x08009e2e)  // lhu id,0(frame); 对 ID 做二分查找(BST)
        │  命中 → thunk(0x0800a1xx)  // 设 a0=帧指针, a1=状态表索引, a2=源指针
        ▼
解码器三类：
   (A) STATE  j 0x08000272   // 把 12 字节整帧 memcpy 进“信号状态表”，按 idx×12 定位
   (B) custom 大函数          // 直接逐位拆 D0..D7，写多个状态变量
   (C) TXINJECT j 0x08008062 // 收到→改写→重算计数器/CRC→用原 ID 回注(见 §4)
        │
        ▼
信号状态表 @RAM(≈0x20002xxx, 12B/条)  // signal_state(dir,idx,buf): a0≠0 读 / a0=0 写
        │
        ▼
打包器：手机发 tx(0xB0/0xD0…) → 读状态表多条 → 位压缩成 33B/29B 包 → ble_notify(0x08000c02)
        │
        ▼
小程序 gaugeParser/batteryParser 逐位解码+定标 → 展示页
```

### 1.1 帧结构实证（D0 = 帧偏移 3）✔FW
`decode_dispatch` 用 `lhu a5,0(s0)` 取 ID（偏移 0，2 字节）。0xD0 电池打包器读 `state[1]`（=0x132）的
**偏移 3、4** 两字节拼成总电压，而小程序 `batteryParser` 把电池包 `[0..1]×0.01` 解释为总电压——
两端对齐 ⇒ **D0(数据字节0)=帧偏移 3**。故 12 字节帧 = `[IDlo,IDhi, DLC, D0,D1,D2,D3,D4,D5,D6,D7, pad]`。

### 1.2 状态表访问器 `signal_state(0x08000272)` ✔FW
```
idx*0xC 定位；memcpy 指针取自 0x4004c；
a0≠0 → 读(state→buf)；a0=0 → 写(buf→state)；每条固定 12 字节。
```

## 3. 状态包装配（哪个 CAN ID 进哪个字段）✔FW

### 3.1 电池包 0xD0（29B，打包器 0x080097cc–0x080099ba）✔FW
打包器按序 `signal_state(读, idx)` 取这些状态条，再位压缩：

| 读取 idx | 来源 CAN ID | 进入电池包字段（×定标见小程序） |
|----------|-------------|-------------------------------|
| idx1 | 0x132 BMS_hvBusStatus | `[0..1]`总电压×0.01V、`[2..3]`总电流×-0.1A |
| idx10 | (0x3d2/0x312 系) | 充/放电 kWh、温度 |
| idx4 | 0x332 BMS_bmbMinMax | `[18..20]`单体 max/min×0.002V |
| idx5 | (能量/容量系) | 容量/保留电量 |
| idx6 | 0x352 BMS_energyStatus | 出厂/当前容量、剩余 kWh |
| idx3 | 0x292 BMS_socStatus | `[24..25]`SOC |

> 结论：**电池包 = 把 0x132/0x292/0x332/0x352/0x3d2/0x312 等 BMS 帧的原始字节重新位压缩**。
> 因此“读 0x132 的 D0..D1 即总电压×0.01”这类**原始 CAN 解析法**与小程序 `batteryParser` 完全一致。

### 3.2 仪表包 0xB0（33B，打包器 0x08008f44 / 0x08007978）
同理由 0x118(挡位/车速)、0x102/0x103(车门)、0x257(车速)、0x266/0x2e5(驱动单元输出)、0x3fd(AP/限速/盲区)、
0x3fe(刹车温度)、0x20c(HVAC)、0x312/0x33a(电池温/续航) 等状态条位压缩而成；逐字段位布局见
`TESLA_CAN_TSL_REFERENCE.md` §1.3（✔APP）。**那张表的“源 CAN ID”列在此被固件证实**。

## 4. 控制：某个状态怎么发出 CAN 帧 ✔FW

### 4.1 命令分发与三个执行器入口 ✔FW
命令分发在 `0x0800957e` 起，对命令码二分（0xA1/0xA3/0xA7/0xB0/0xC0/0xC1…）。三个控制命令的去向：
```
167(0xA7): lbu a2,0(s1)  // a2 = 动作码 = payload[0]
           a1=0xA7; a0=s0; jal 0x0800b33a      // → 167 动作执行器(跳转表，§4.2)
187(0xBB): a1=0xBB; c.j 0x0800954e             // → 共用执行器(异步置位)
162(0xA2): lbu a2,0(s1); a1=0xA2; c.j 0x0800954e
```
**三命令都只带 1 字节动作码**；固件内部把它翻译成 CAN 注入。**注意：动作执行不是同步构帧，而是
先把"内部动作号"写入 RAM 待执行变量，由主循环异步合成并注入 CAN**（见 §4.5），因此一个动作可能
对应一段 CAN 序列而非单帧。

### 4.2 167 动作执行器 = 跳转表 ✔FW
入口 `0x0800b33a`：
```
a5 = [gp+0x10c]; if([a5+4] & 8) return       // 模块禁用门禁
if (a2 > 0x47) goto default                  // 动作码上限 = 71
a3 = 0x0800ef38                              // 跳转表基址(auipc+addi 实证)
target = a3 + (int32)tableEntry[a2]          // tableEntry = u32×72
jr target                                    // 进入该动作码的 case
```
每个 case 极短：`c.li a0,<内部动作号>; jal/c.j <处理函数>`。即 **167 动作码(0..71) → 内部动作号 → 处理函数**。
72 项跳转表落在 `0x0800b36e..0x0800b510`，汇聚到 44 个处理函数。下表合并：动作码、内部号(a0)、
小程序语义（✔APP `vehicleControl.js`/`commandSafety.js`）：

| 动作码 | 内部号a0 | 含义(✔APP) | 动作码 | 内部号a0 | 含义(✔APP) |
|--------|---------|-----------|--------|---------|-----------|
| 0 | 1 | 左前门 | 16 | 0 | 后视镜折叠 |
| 1 | 4 | 右前门 | 18/19 | 0/1 | (车身) |
| 2 | 7 | 左后门 | 21/22/23 | 2/1/3 | (车身) |
| 3 | 10 | 右后门 | 24/25 | 0/1 | (车身) |

### 4.3 注入封装 `tx_wrapper(0x08008062)` ✔FW
```
jal 0x080001d4          // CAN_TX 门禁(LISTEN_ONLY)：为 0 直接 return，不发
StdId = frame[0](u16)   // 发送 ID 由帧结构体决定
DLC   = frame[2]
memcpy(TxMsg.Data, frame+3, DLC)
CAN_Transmit(CAN1=0x40006400)
```

### 4.4 “收到→改写→原 ID 回注”(re-sign) ✔FW（实证代码）
TXINJECT 型 ID 把收到的帧改几个数据位、重算特斯拉滚动计数器+CRC、用**原 ID** 发回。
实证片段（处理函数 `0x08007c16`，反汇编）：
```
sb   0x0a, 3(s0)        // D0 = 0x0a        ← 改写数据位
lbu  a5,   9(s0)        // 读 D6（counter 字节，帧偏移9）
addi a5, a5, 0x10       // counter += 0x10  （高 nibble +1）
sb   a5,   9(s0)        // 写回 D6
srli a5, a5, 4          // counter >> 4 作 CRC 表索引
add  a5, a5, CRC_table  // CRC_table ≈ 0x0800eb00 区(auipc 实证)
lbu  a5, 0(a5)          // CRC = table[counter>>4]
sb   a5, 0xa(s0)        // D7 = CRC（帧偏移10）
c.j  0x08008062         // → tx_wrapper（原 ID 回注）
```
> 帧布局下 **偏移9=D6=滚动计数器、偏移10=D7=CRC**（修正旧文 frame[9]/[0xa] 的字节名）。

re-sign 处理函数各自改写的数据字节（✔FW 抽取）：

| 处理函数 | 改写数据位 | 发送点 | 关联 ID(◎) |
|----------|-----------|--------|-----------|
| 0x08007c16 | D0=0x0a | 0x08007c72 | 0x334/事件 |
| 0x08007c7c | D5=0xc8 | 0x08007cfa | — |
| 0x08007d04 | D0=0x02 | 0x08007d3e/0x08007d58 | — |
| 0x08007d5c | D2=0x30, D0/D1=动态 | 0x08007dc0 | — |
| 0x08007e68 | D1 改 + 计数器/CRC | 0x08007ef4 | 0x3b0/0x4a8 注入入口 |

回注/事件 ID 集合（✔FW）：解码 thunk 尾跳 tx_wrapper 的 `0x082 0x1f9 0x229 0x249 0x293 0x2b6 0x333 0x3c3 0x679`
+ 事件短表 `0x189 0x21c 0x334 0x3a1 0x3b0 0x3e9`（@`0x0800f17a`）。

### 4.5 全部 CAN 发送点（32 处）✔FW
全镜像扫描 `tx_wrapper(0x08008062)` 的调用点共 **32 处**（旧文档说 18，实测更多）：
```
0x0800267c 0x08002712 0x08002a82 0x08002ba8 0x080032fe 0x08003b72 0x08003c50 0x08003cdc
0x08003e6a 0x080043f6 0x08004778 0x08004852 0x0800485e 0x08004cc0 0x08004cc6 0x080050e4
0x08006294 0x0800631a 0x0800686a 0x08006dd2 0x08006f42 0x0800718e 0x08007466 0x08007c72
0x08007cfa 0x08007d3e 0x08007d58 0x08007dc0 0x08007ef4 0x08008372 0x08009a80 0x0800c68c
```
帧构造分三类（✔FW 实样）：
- **改写回注**：栈/RAM 里已收到的帧，改几位+重签名（§4.4），用原 ID 发。
- **状态转发**：先 `signal_state(读,idx,&frame)` 把某状态条读成帧再发（如 `0x08003cdc` 前调 `0x08000272`）。
- **模板构帧 / BLE→CAN 透传**：`0x08009a80` 实证（✔FW）——手机发 `[idHi,idLo,data…]`，固件直构帧：
  ```
  frame[0..1] = (payload[0] << 8) | payload[1]   // CAN ID = BLE payload 前两字节(大端)
  frame[2]    = len                              // DLC
  memcpy(frame+3, payload+2, len)                // 数据 = payload[2..]
  tx_wrapper(frame)                              // 发任意 ID 的 CAN 帧
  ```
  即**固件支持手机直发任意 CAN 帧**；小程序 `commandSafety` 白名单未暴露此命令，属底层调试能力。
- **状态转发**：`0x08006294` 实证——`signal_state(读, idx=8, &frame)` 读某状态条→改几位→发回。
> 其余发送点的 frame[0](ID) 多来自栈缓冲/状态拷贝/模板，需逐点反汇编（路线见 §8）。

### 4.6 动作码语义字典（✔APP 反推补充）
167/187/162 的动作码"是什么意思"由小程序补全（固件只存号、不存名）：
- `162` 滚轮：1上/2右/3下/4左/5确认/6长按（`vehicleControl.js`）。
- `167` 直接车控：门 0-3、电池加热 4/5、备箱 8/9、副驾座椅 12-15、后视镜 16、主驾座椅 26-29、记忆 52-71。
- `187` 执行表(`functionOptions.js` `fun`，~200 项)：座椅/车门/AP/灯光/喇叭/模块控制(250 启停/253 重启)/4G/外接屏…
> 这些是“**动作的人类语义**”；与 §4.2 的“内部动作号”一起，构成重写固件时动作子系统的行为规格。

### 4.7 全部 32 发送点的改写映射（✔FW 批量，`../../scripts/fwsend.py`）
**核心结论**：除 BLE 透传(`0x08009a80` 构造新 ID)外，**31 个发送点全是"改写收到帧的数据位 + 原 ID 回注"**。
即**固件控制车辆 = 改写特斯拉对应控制帧的特定位**（再重签名回注），**不构造全新帧**。各点触发状态+改写位：

| 发送点 | 触发 gp 状态 | 改写数据位 |
|--------|------------|-----------|
| 0x0800267c | gp+0x165 (0x2b6 触发) | D0 \|= 0x48 |

### 4.8 主分发器结构（✔FW `0x08009e2e–0x0800a720`）
整个接收处理由一个**主分发器**驱动，`lhu a5,0(s0)` 读 frame[0]=ID，再 `beq/bne a5,a4` 与 ID 常量比较路由：
- `0x08009e2e–0x0800a146`：ID→handler 比较表（`fwall.py` 解析，53 监控 ID）。
- `0x0800a146–0x0800a21a`：**perid 解析分发**——尾跳转到各解析函数
  （j→`0x08004722`TRIP / `0x08004f7a`VCRIGHT_door / `0x08004370`SCCM_lever / `0x08002946`UI_status / `0x08004f26`VCFRONT …）。
- `0x0800a21a–0x0800a720`：**re-sign 改写分发**——按 ID 路由到改写函数（见 §4.10）。

## 5. 原始 CAN 解析算法速查（不经 TSL 模块，直接读总线）

把上面 ✔FW 的字节访问 + ✔APP 的定标合并，即“直接从特斯拉总线解析”的配方：

```
0x132 BMS_hvBusStatus:  电压 = (D0|D1<<8)×0.01 V ;  电流 = (D2|D3<<8)×-0.1 A
0x292 BMS_socStatus:    SOC  = (取 7/10bit 字段)
0x332 BMS_bmbMinMax:    单体max/min = 12bit×0.002 V
0x352 BMS_energyStatus: 出厂/当前容量 ×0.1 / ×0.02 kWh
0x3d2 BMS_kwhCounter:   累计充/放电 = u32×0.001 kWh
0x401 BMS_brickVoltages: D0=mux索引; 每帧 3×u16×0.0001 V (收齐 mux 拼 96 串)
0x118 DriveSystemStatus: 挡位 = D2[5:7]→0/P/R/N/D
0x257 DI_speed:         车速 = D3..D4 拼 9bit
0x102/0x103 门状态:     D0/D1 位 → 左前/左后/右前/右后/前后备箱
0x3fe 制动:            4×刹车温度 = D1/D2/D5 的 >>6 等 10bit 字段 -40 ℃
0x321 VCFRONT_temp:    环境温度 = D5×0.5 - 40 ℃
0x3fd AP/DAS:          AP状态/道路限速(×5)/盲区/手扶 位字段
```
通用换算：有符号 n 位 `x≥2^(n-1)?x-2^n:x`；温度多为 `×0.25-25` 或 `×0.5-40`；功率 11bit 有符号 `/2` kW。

## 9. 关键结论
1. **解析法**：`decode_dispatch(0x08009e2e)` 按 ID 二分→解码器把 D0..D7 拆位或整帧入状态表→打包器位压缩成
   29B/33B 包；原始 CAN 字节→信号的定标与小程序解析器逐项吻合（§5）。
2. **控制法**：167/187/162 只带 1 字节动作码→（167 跳转表/共用执行器）→内部动作号→**异步**合成 CAN；
   TXINJECT 型对收到帧做"改数据位+滚动计数器(D6)+CRC(D7)+原 ID 回注"（§4.4）。
3. **进度**：解析架构、53 ID、40 解码器字节访问、电池包来源、167 跳转表、re-sign、32 发送点 —— 均已实证；
   逐位精确公式与逐动作码 CAN 帧按 §8 路线逐项补全。
