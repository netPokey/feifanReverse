# CH32V208 蓝牙 CAN 收发器固件 — 深度逆向分析（v2）

> ⚠ **命名校验提示**：本文部分 CAN ID 命名（0x118/0x266/0x293/0x2f3/0x31f/0x339 等）为校验前旧版。
> **命名以 [`TESLA_CAN_OFFICIAL_NAMES.md`](./TESLA_CAN_OFFICIAL_NAMES.md) 为准**；结构/例程分析仍有效。导航见 [`README.md`](./README.md)。
>
> 固件：`c6eec2f9-firmware.bin`（62 464 B / 0xF400，裸二进制，无符号）
> 平台：WCH **CH32V208**（青稞 V4，RV32IMAC）+ `CH32V20x_BLE_LIB_V1.3` + bxCAN
> 本文给出：① 复现方法 ② 整体结构 ③ 全部 ID 逐条标注 ④ 解码例程 ⑤ 注入/发送例程
> ⑥ 蓝牙控制层 ⑦ 安全提示。所有结论都带**地址级证据**，可按第 1 节复现。

## 1. 复现方法与环境

```bash
# 依赖：python3 + capstone（带 RISC-V）
pip3 install capstone            # 5.0.9 起内置 CS_ARCH_RISCV / CS_MODE_RISCVC

# 关键参数：Flash 基址 0x08000000（复位向量 j 0x0800d034 + 外设 0x40006xxx 印证）
```

核心反汇编脚本（线性扫描，自动识别 2/4 字节指令）：

```python
import struct, capstone
from capstone import Cs, CS_ARCH_RISCV, CS_MODE_RISCV32, CS_MODE_RISCVC
data=open("firmware.bin","rb").read(); BASE=0x08000000
md=Cs(CS_ARCH_RISCV, CS_MODE_RISCV32|CS_MODE_RISCVC)   # 必须带 RISCVC 才能解压缩指令
addr=BASE; off=0
while off < len(data):
    i=next(iter(md.disasm(data[off:off+4], addr, count=1)), None)
    sz=i.size if i else 2
    if i: print(f"0x{i.address:08x}: {i.mnemonic} {i.op_str}")
    addr+=sz; off+=sz
```

> 注意：capstone 对 `jal/auipc/lui` 的立即数显示不可靠，跳转/分支目标需**自行从原始
> 指令位重算**（J-type / B-type / CJ 立即数）。本文所有“目标地址”均为重算值。

分析步骤（每步对应一节）：
1. **定位常量表**：把 bin 当 u16 LE 扫描，找“连续且全部 ≤0x7FF”的区段 → CAN ID 表（§2.2）。
2. **识别 bxCAN 库函数**：搜 `0x40006400` 基址 + 偏移 `0x180/0x1b0/0x200/0x240` → 收/发/过滤器（§2.1）。
3. **建调用图**：重算所有 `jal/c.j` 目标，做 xref → 找发送/过滤器/接收的调用者。
4. **解 ID 分发器**：在 `0x08009e2e` 的二分查找里抓 `addi a4,ID; beq/bne` → ID→例程（§3）。
5. **顺藤摸瓜**：每个 ID 例程是 thunk，尾跳 `j` 到真正解码函数（§3 表）。
6. **蓝牙层**：从 `FilterSetup`/`TX_wrapper` 的调用者回溯到 GATT 写回调与 notify（§6）。

## 2. 固件整体结构

### 2.1 关键函数地址（裸机，地址即证据）

| 地址 | 名称 | 说明 |
|------|------|------|
| `0x0800d034` | Reset/`_start` | `auipc/addi gp`（gp=0x27ffc000）、初始化 |
| `0x0800d526` | `CAN_FilterInit` | 写 `FMR(0x200)/FS1R(0x20c)/FM1R(0x204)/FFA1R(0x214)/FA1R(0x21c)`、组寄存器 `0x240/0x244` |
| `0x0800d628` | `CAN_Transmit` | 取空邮箱，`StdId<<21`→`TIxR(0x180)`，DLC/数据→`TDTR/TDLR/TDHR` |
| `0x0800d6fc` | `CAN_Receive` | `RIxR(0x1b0)>>21` 取 StdId，读数据并释放 FIFO |
| `0x08008062` | `tx_wrapper(frame)` | **发送门禁**`0x080001d4`→为 0 跳过；ID 取自 `frame[0]`（u16）。**18 处调用** |
| `0x08008116` | `filter_load(ids,len)` | 拷 ID 数组（≤112B=56×u16）→ **14 组×4 ID 的 16-bit 列表模式**（`id<<5`） |
| `0x080080ce` | `filter_acceptall()` | 14 组 mask=0，**全通嗅探** |
| `0x080080a8` | `filter_single()` | 单组过滤器 |
| `0x0800825e` | `CAN_RX_ISR` | `CAN_Receive`→`rx_store(id,dlc,data)`→转发蓝牙（`mret` 结尾，是中断） |
| `0x08008218` | `rx_store` | 存入 12B 步长 RAM 帧表，并经函数指针外送 |
| `0x08000272` | `signal_state(dir,idx,buf)` | **解码信号状态表**访问器：RAM 表，**12B/条**，按 idx 定位；dir=0 读/否则写 |
| `0x080001a6` | `scale_helper` | 取值×10÷16，供多个解码器换算后写状态 |
| `0x08000c02` | `ble_notify(conn,type,buf,len)` | GATT 通知，len≤0x40F；conn 取自 `gp+0x142` |
| `0x08009e2e` | `decode_dispatch(frame)` | **解码总分发器**：`lhu id,0(frame)` 后二分查找（§3） |
| `0x08007f8e` | `event_dispatch(frame)` | 6 个 ID 的**事件检测**分发（布尔检测+计数，§5.3） |

### 2.2 三张常量表

| 表 | 地址 | 内容 |
|----|------|------|
| **监控 ID 表** | `0x0800f188` | 53 个有效 11-bit ID + `0x7ff` 哨兵（§3 全表） |
| **注入/事件短表** | `0x0800f17a` | `0x334 0x3e9 0x21c 0x189 0x3a1 0x3b0`（6 个），前缀 RAM 指针 `0x2000412c` |
| **周期/超时元数据** | `0x0800ea48` | 成对 (周期ms, 超时ms)：50/150、40/96…用于丢包/周期性分析 |
| **CRC 查找表** | `~0x0800eadc` | 注入时重算特斯拉校验字节用 |

### 2.3 数据流水线

```
车辆 CAN 总线
  └─(bxCAN 硬件过滤: 全通 或 0x0800f188 白名单 或 蓝牙下发)
      └─ CAN_RX_ISR(0x0800825e) ─ CAN_Receive ─ rx_store(0x08008218)
            ├─ decode_dispatch(0x08009e2e): 按 ID 解码 → 写信号状态表(0x08000272, 12B/条)
            ├─ event_dispatch(0x08007f8e): 6 个 ID 做事件检测/计数
            ├─ 个别 ID(0x333/0x4f3/0x3b0/0x4a8…): 改写+重签名 → tx_wrapper 回注
            └─ 原始帧/状态 → ble_notify(0x08000c02) 推送手机
手机 APP ──(GATT 写)──> ble_cmd 处理器(0x080097xx):
            ├─ 0xD0: 读状态表打包 29B 状态包并 notify
            ├─ 0xFF 0xFF + idlist: 重设 RX 过滤白名单(filter_load)
            ├─ 帧(id+data,len≤8): 发送一帧 CAN(tx_wrapper)
            └─ 0xC0/0xC1/0xBB: 模式/开关控制
```

## 4. 典型解码例程详解

解码器有两种实现：

**(A) 走通用状态表 `0x08000272`**（如 0x132/0x292/0x332/0x352/0x3d2…）
```
thunk: 设置 a0=方向, a1=信号索引, a2=源数据指针 → j 0x08000272
0x08000272: 取 memcpy 指针(0x4004c); a5 = RAM表基 + idx*0xC;  // 每条 12 字节
            dir==0: memcpy(buf, &table[idx], 12)  读
            else  : memcpy(&table[idx], buf, 12)  写
```
即把该 ID 的解码结果以 **12 字节/条**写入 RAM 信号表；蓝牙 0xD0 再按索引读出（§6）。

**(B) 走换算器 `0x080001a6`**（如 0x266/0x312/0x321/0x33a/0x31f/0x3b6…）
```
0x080001a6: r = (*fn_0x4007c)();  r = r*10>>4;  // 0.625 类比例换算后入状态
```
对应仓库文档里 `UI_ratedConsumption ×0.625 Wh/km`、温度 `×0.5/0.25` 这类定标。

**(C) 独立大函数**（如 0x118 `0x080056a8` 写 3 个状态位、0x3fe `0x08004b5e` 写 4 个、
0x401 `0x080090dc` 处理 mux 多路复用单体电压、0x257/0x25d `0x0800b53a`）——逐位拆 8 字节
负载，写多个 `gp+offset` 状态变量。例如仓库文档对 `0x401`：Byte0=mux 索引，每帧 3 个 16-bit
单体电压，需收齐所有 mux 帧得到 96 串。

## 5. 【发送 / 注入】例程详解

所有发送都先过门禁 `0x080001d4`（默认只听 / LISTEN_ONLY，与仓库 README 安全策略一致）。

### 5.1 统一发送封装 `tx_wrapper`（`0x08008062`）
```
0x0800806a jal 0x080001d4         // CAN_TX 允许？
0x0800806e if(!ok) return         // 门禁
0x08008070 StdId = frame[0]       // 发送 ID 来自帧结构体，不是写死
0x08008074 DLC   = frame[2]
0x08008090 memcpy(TxMsg.Data, frame+3, DLC)
0x0800809c CAN_Transmit(CAN1=0x40006400, &TxMsg)
```
→ **发送 ID 完全由调用方/蓝牙指令决定**。

### 5.2 拦截—改写—原 ID 回注（re-sign）
处理函数（`0x08007c16/0x08007c7c/0x08007cfa/0x08007d3e/0x08007d58/0x08007dc0/0x08007ef4`
等约 7 个）模式一致：
```
查门禁/限速计数器 → 改某些数据位 →
  frame[9] = (frame[9]+0x10)        // 特斯拉 4-bit 滚动计数器(counter)
  frame[0xA]= CRC_table[counter>>4] // 重算校验字节(查表 ~0x0800eadc)
  → c.j tx_wrapper                  // 用原 ID 重发
```
解码器在 `0x0800a4e4` 明确把 **`0x3b0`(944)、`0x4a8`(1192)** 交给注入入口 `0x08007e40`；
另外 **`0x333`(819)、`0x4f3`(1267)** 的解码 thunk 直接尾跳 `tx_wrapper`（§3 表标 RE-SIGN+TX）。
> 与仓库 ESP32 版 `processIsaChimeSuppression()` 改 `0x399` 数据位+重算校验后回注是同一手法。

### 5.3 事件检测分发器 `0x08007f8e`（不发送，属分析）
对 `0x334/0x189/0x21c/0x3b0/0x3e9/0x3a1` 六个 ID 调用布尔检测回调
（`0x0800799e…0x08007a52`：查 byte3/4/5/7/8/0xA、累加校验等），命中则对 `gp+0x1de..0x1e3`
计数——用于“按下/状态翻转”这类**事件统计**。

### 5.4 发送 ID 小结
- **改写+原 ID 回注**：`0x333 0x3b0 0x4a8 0x4f3`（+ 5.2 系列处理函数覆盖的帧）。
- **蓝牙透传任意 ID**：APP 下发 `[id_hi,id_lo,data...]` → tx_wrapper（§6）。
- 全程仅 1 个 `CAN_Transmit`，被 18 处调用，无固定周期广播表。

## 6. 蓝牙控制层（命令 + 推送）

GATT 写回调把负载 `s1`、长度 `s2`、连接上下文 `s0`、命令码 `a1` 送入分发器
（`0x080097xx`）。

### 6.1 手机→设备 命令

| 命令码`a1` | 触发条件 | 行为 | 证据地址 |
|------|------|------|------|
| `0xD0` | — | 读状态表打包 **29B(0x1d) 状态包**并 notify（§6.3） | `0x080097cc` |
| `0xC0` | 子命令 `s1[0]`∈{0,1,2} | 模式/开关控制（调 `jal -0x6fc` 等） | `0x08009788` |
| `0xC1` | — | 控制/查询 | `0x08009774` |
| `0xBB` | — | 控制/查询 | `0x08009780` |
| `0xFF 0xFF`+idlist | `s1[0]=s1[1]=0xFF` | **重设 RX 过滤白名单**：`filter_load(s1+2, s2-2)` | `0x08009a04→0x08008116` |
| 帧(len≤8) | 长度≤8 | **发送一帧 CAN**：id=`s1[0..1]`，dlc=len，data=`s1[2..]`→`tx_wrapper` | `0x08009a3a→0x08008062` |
| 单组过滤 | — | `filter_single()` | `0x080099f6→0x080080a8` |

> 即手机可：① 自定义采集哪些 ID（动态白名单）② 让设备在总线上发任意 ID 的帧
> ③ 切换工作模式 ④ 拉取整车状态。

### 6.2 设备→手机 通知类型
notify 统一走 `ble_notify(conn, type, buf, len)`（`0x08000c02`）。已见类型码：

| type | 长度 | 内容 |
|------|------|------|
| `0xD0` | 29B | 打包整车/电池状态（§6.3） |
| `0xC1` | 4B | 计数/帧回执 |
| `0xBB`/`0xBC` | 1B | 状态/ACK |
| (流) | 12B/帧 | `rx_store` 经函数指针把收到的原始帧外送 |

### 6.3 0xD0 状态包打包过程（`0x080097cc–0x080099ba`）
依次 `signal_state(读, idx, tmp)` 取多个解码信号（索引 1/3/4/5/6/0xA…），再用大量
移位/掩码压进 29 字节：典型字段宽度 `andi 0x3ff`(10-bit，SOC/电压类)、`0x7f`(7-bit，温度/
百分比类)、`0x3f`(6-bit)，与仓库文档里 `BMS_socUI(10bit)`、`UI_actualSOC(7bit)`、
`packT(9bit)` 等定义吻合。打包好后 `ble_notify(s0, 0xD0, buf, 0x1d)` 推送。

> 📎 **深化（小程序侧地面真值）**：本节命令码 `a1` = 小程序 BLE 帧 `type` 字段；完整命令字典
> `160–240`、各 payload 逐位布局、鉴权握手 `168`、OTA 链路，见
> **`FIRMWARE_CH32V208_BLE_PROTOCOL_DEEP_ANALYSIS.md`**（小程序 × 固件 联合逆向）。
> 其中本节 `0xD0` 的 29B 包被精确命名为 **`208` 电池概览**，字段与 `services/batteryParser.js` 逐项吻合。

## 7. 安全提示
该固件具备在车辆总线上**伪造/改写并回注帧（含 BMS、驱动、UI 相关 ID）**的能力，且蓝牙
可命令其发送**任意 CAN ID**。务必：
1. 仅在**隔离测试台**、确认 `CAN_TX` 门禁关闭时上电观察；
2. 未经测试台验证，不要接入真实车辆总线；
3. 蓝牙命令面无鉴权（命令码即明文），属调试形态，量产需加认证。
