# 特斯拉 CAN ↔ TSL 模块 — 实战参考手册（读状态 / 控功能 / 全算法）

> 目标：把“**怎么从 CAN 读到特斯拉车机的各种状态**”和“**怎么通过蓝牙控制 TSL 模块的各项功能**”
> 一次讲透——含**数据流、字段位布局、定标算法、功能码表、发送/注入算法**。
> 来源：小程序 `weapp-rebuild/`（解析/控制的地面真值）× 固件 `firmware.bin`（CAN 收发/注入）。
> 证据：小程序给 `文件:行`；固件给**地址**或**实测**（本文 ID 表是从 bin 直接扫描所得，可复现）。
>
> ⚠️ 仅用于隔离测试台/学习。注入特斯拉总线有风险，务必确认固件 `CAN_TX` 门禁状态。本文档不入库。

---

## 0. 一图看懂两条链路

```
【接收链路：读状态】
 特斯拉车辆 CAN 总线
   │  (固件硬件过滤：全通 或 53-ID 白名单 @0x0800f188)
   ▼
 固件 CAN_RX_ISR(0x0800825e) → 每个 ID 专用解码器 → 写“信号状态表”(0x08000272, 12B/条)
   │
   ├─ 手机发 tx(0xB0=176,[1]) 开启仪表轮询 → 固件打包 33B 仪表帧 → notify type=176
   ├─ 手机发 tx(0xD0=208,[1]) 开启电池轮询 → 固件打包 29B 电池帧 → notify type=208
   ├─ tx(0xD1=209)/tx(0xD2=210) → 电芯/DCDC 帧
   ▼
 小程序 onPacket(type) → gaugeParser/batteryParser 逐位解码+定标 → setData → 展示页

【发送链路：控功能】
 展示页点击（开门/座椅/灯光/AP…）
   ▼
 小程序 tx(type, [动作码])     // type=0xA7/0xBB/0xA2…；动作码来自功能表
   │  commandSafety.js 白名单校验（页面来源/长度/权限/硬件）
   ▼  0x55 0x7F <type> <lenHi> <lenLo> <payload...> <校验>
 固件 GATT 写回调 → 命令分发(0x080097xx) → 选定目标 CAN ID + 改数据位
   → 重算滚动计数器(byte9)+CRC(byte10) → tx_wrapper(0x08008062) 注入 CAN 总线
```

**一句话**：读状态 = 订阅固件聚合的 `176/208/209/210` 包 + 套用本文解码算法；
控功能 = 发 `type+动作码`，固件替你合成并注入特斯拉 CAN 帧。

## 1.5 电芯(209)/DCDC(210)

- **电芯 mux（209，8B）** `batteryParser.js:96`：`[0]`=mux 索引 → 实际电芯号 `=index×3`；
  `[2..7]` = 3 个 u16 `×1e-4 V`。要收齐全部 mux 帧才能拼出整组电芯（最多 96 串）。
  → 对应固件 `0x401 BMS_brickVoltages`（mux 多路，解码 `0x080090dc`）。
- **DCDC（210，8B）** `batteryParser.js:131`：输入电压 `[2..3]×0.1`、输出电压 `[0..1]×0.01`、
  输出电流 `[4..5]×0.1`、输出功率=`Vout×Iout`。

## 1.6 设备信息 + 设置位（160/163, ≥18B）

`device.js:52` + `settingsOptions.js`：`[0]`hw_ver、`[1]`bt_ver、`[2]`fw_ver、`[4..15]`id(24hex)、
`[12..17]`mac(逆序)、`[40]==1`→`[41..58]`VIN；**`[24..34]` 为模块设置位域**（§2.6）。

## 1.7 状态变化事件（展示页/自动化用）

`gaugeEvents.js:47` 在两帧仪表之间做**边沿检测**，产出事件（供联动/超级快捷/自动化触发）：
车门 0→1 开、前/后备箱开、挂 D/挂 P、AP 变化、灯光变化、盲区出现、手扶提醒出现。
算法即“前一帧该位=0 且当前帧=1”。

# 第三部分 · 核心算法集

## 3.1 BLE 帧封装 / 拆解（双向同构，`ble.js:129,191`）

```
帧 = 0x55 0x7F  type  lenHi lenLo  payload[len]  checksum
checksum = (Σ 从 type 到 payload 的所有字节) & 0xFF
```
- 发送：拼帧→算校验→按 `min(MTU,247)-3` 分片逐包写 `FFF1`。
- 接收：状态机找 `0x55→0x7F`→type→len(大端)→收 payload→比对校验；包间隔 ≥300ms 复位防粘连。
- GATT：Service `FFF0`、读写+通知特征 `FFF1`。

## 3.2 解码定标公式（读状态用）

```js
// 小端取整数
u16 = b[i] | b[i+1]<<8
u32 = b[i] | b[i+1]<<8 | b[i+2]<<16 | b[i+3]<<24
// 有符号补码（n 位）  —— 电机功率/海拔用
sign(x,n){ return x >= 1<<(n-1) ? x-(1<<n) : x }
// 常用定标
电压    = u16 × 0.01            // V (0x132)
电流    = u16 × -0.1            // A，<-3276.7 时 +6553.6
单体电压 = field12 × 0.002       // V (0x332/0x401)
电芯电压 = u16 × 0.0001          // V (0x401 mux)
温度    = field × 0.25 - 25      // ℃ (0x312)  或 ×0.5-40 (环境)
续航    = field10 × 1.61         // mi→km (0x33a)
里程    = field26 / 10           // km
电机功率 = sign(field11,11) / 2   // kW (0x266/0x2e5)
胎压    = byte × 0.025           // bar
能量    = u32 × 0.001            // kWh (0x3d2)
```

## 3.3 注入/重签名算法（控功能时固件做，源在 `firmware.bin`）

手机发 `167/187/162 + 动作码`，固件据码选定**目标特斯拉 CAN ID**并改数据位，然后**必须重算**
特斯拉帧尾的滚动计数器与校验，再用**原 ID** 注入（DEEP_ANALYSIS §5.2）：

```
frame[9]  = frame[9] + 0x10            // 4-bit 滚动计数器(counter)，高 nibble +1
frame[10] = CRC_table[counter >> 4]    // 查表重算 CRC 字节(表 ~0x0800eadc)
tx_wrapper(frame)                       // 取 frame[0] 为 StdId 注入；先过 CAN_TX 门禁 0x080001d4
```
- 已确认的“改写+原 ID 回注”目标：`0x333 UI_chargeRequest`、`0x3b0`、`0x4a8`、`0x4f3`，
  以及事件/注入短表 `0x189 0x21c 0x334 0x3a1 0x3b0 0x3e9`。
- **门禁**：`0x080001d4` 默认 LISTEN_ONLY；门禁为 0 时所有注入被跳过（只听不发）。
- ⚠️ **逐个动作码 → 确切 CAN 帧字节** 的全表在固件内部 switch 里，本手册给出**机制与目标 ID**，
  未逐码展开（需进一步反汇编 167/187 handler；可用已装好的 capstone 5.0.7 续做）。

## 5. 落地用法小结

**读状态（3 步）**：① `tx(0xA8,"1234")` 鉴权 → ② `tx(0xB0,[1])`/`tx(0xD0,[1])` 开轮询 →
③ `onPacket(176/208…)` 收帧，套 §1.3/§1.4 算法解析。

**控功能（2 步）**：① 鉴权 → ② `tx(0xA7,[码])` 直接车控 或 `tx(0xBB,[码])` 执行功能表动作
（码见 §2.2/§2.3）。注入是否真正上总线，取决于固件 `CAN_TX` 门禁。
